# Desktop Review Design

**项目**: `memo-inbox-client`
**日期**: `2026-04-13`
**范围**: `apps/desktop` 回顾页与顶部导航切换
**状态**: 已确认设计，待用户审阅

## 1. 背景

当前桌面端已经具备：

- 收件流首页
- memo 创建
- memo 列表浏览
- 基础搜索与“今天 / 有图片”筛选
- 顶部导航视觉位：`全部 / 回顾 / 归档`

但当前“回顾”仅有导航文案，没有真实路由与页面能力。

后端当前可用于回顾的接口只有：

- `GET /review/random`
- `GET /review/daily`

其中当前实现存在明确边界：

- `review/random` 不是严格随机，而是返回最近列表中的首条
- `review/daily` 不是“今日回顾”，而是返回最近 100 条里最早的一条

因此桌面端不能把“回顾”设计成复杂回顾系统，也不能把 `daily` 包装成真正有意义的“每日回顾”。

## 2. 目标

为桌面端提供一个独立的“回顾”主视图，服务于单一目标：

- 重温一条旧 memo

最小闭环为：

- 顶部导航可以在 `全部` 与 `回顾` 之间真实跳转
- 当前页面有明确选中态
- 用户进入 `回顾` 页时自动加载一条旧 memo
- 用户可以点击一个主操作重新获取一条旧 memo

## 3. 非目标

本次明确不包含以下能力：

- 每日回顾模式
- 多策略回顾
- 真随机抽样
- 回顾历史
- 收藏 / 跳过 / 不感兴趣
- 搜索与回顾混合页
- 多条推荐列表
- 归档页实现
- 前端补偿去重或伪随机策略

## 4. 方案选择

### 方案 A：独立的旧记忆页

这是本次采用方案。

做法：

- 新增 `/review` 路由
- 页面只展示一条当前 memo
- 唯一主操作是“换一条”
- 数据直接复用 `review.random`

优点：

- 与当前后端能力最一致
- 页面目标单一，符合 KISS
- 不与首页搜索和“今天”筛选重复
- 不会把轻量接口包装成复杂系统

### 方案 B：双入口回顾页

不采用。

做法是同时暴露 `random` 和 `daily`。

原因：

- `daily` 当前语义和“每日回顾”名称不一致
- 容易让用户误解为系统具备稳定回顾策略

### 方案 C：回顾 + 搜索混合页

不采用。

原因：

- 会和首页已有浏览 / 搜索能力重叠
- 页面焦点会从“重温一条旧记忆”变成“继续检索”

## 5. 信息架构

### 5.1 顶部导航

桌面端顶部导航继续保留：

- `全部`
- `回顾`
- `归档`

行为定义：

- `全部` 对应 `/`
- `回顾` 对应 `/review`
- `归档` 当前无后端支持，只保留导航位，不进入可用页面

状态定义：

- 当前路由项必须有明确选中态
- 其他可用项保持未选中态
- `归档` 需要弱化表现，不伪装成已接入功能

### 5.2 回顾页定位

`/review` 是一个独立模式页面，不是首页局部插槽。

它的页面任务只有一个：

- 从现有 memo 中重新看一条旧记录

因此页面不承载：

- 新建 memo 输入框
- memo 流式列表
- 搜索入口
- “今天”筛选复刻

## 6. 交互设计

### 6.1 初始进入

- 用户点击顶部 `回顾` 导航进入 `/review`
- 页面首次挂载时自动请求 `GET /review/random`
- 返回的一条 memo 作为当前页面主内容

### 6.2 主操作

页面只有一个高优先级动作：

- `换一条`

点击后行为：

- 再次调用 `GET /review/random`
- 用新返回结果替换当前主卡片内容

文案约束：

- 不使用“随机一条”
- 不使用“每日回顾”
- 不使用“智能推荐”

原因：

- 当前接口不支持这些更强语义

### 6.3 同条重复结果

如果后端连续返回同一条 memo：

- 前端直接展示返回结果
- 不增加客户端去重、缓存轮换或伪随机逻辑

原因：

- 不能通过前端补偿制造后端不存在的回顾策略

## 7. 页面结构

### 7.1 整体骨架

Desktop 继续保持现有结构：

- 统一 header
- 左侧主内容
- 右侧 quiet rail

### 7.2 左侧主内容

左侧主列包含：

- 页面标题
- 一句简短副标题
- 当前 memo 主卡片
- “换一条”主操作

标题方向：

- `重温旧记忆`
- 或 `旧记忆`

副标题方向：

- 克制、安静、说明这是重新翻看一条旧 memo 的地方

### 7.3 当前 memo 卡片

主卡片沿用现有桌面卡片语言，不新发明视觉体系。

内容结构：

- 记录时间
- 正文内容
- 图片附件
- 标签

不新增：

- 推荐理由面板
- 情绪分析
- 自动摘要
- 时间轴关系

### 7.4 Quiet Rail

右侧只放低密度辅助信息，建议限制在两块内：

- 回顾说明
- 当前 memo 的基础上下文

上下文只能使用当前真实字段，例如：

- 创建时间
- 标签数量
- 是否包含图片

不展示伪统计、伪趋势或复杂运营面板。

## 8. 状态设计

### 8.1 加载中

- 主内容区显示一张与 memo 卡片结构相近的骨架
- 保持当前页面留白和安静节奏

### 8.2 空状态

如果接口返回 `MEMO_NOT_FOUND`：

- 页面显示“还没有可回顾的记忆”这一类直接文案
- 提供返回 `全部` 页的入口

### 8.3 失败状态

如果请求失败：

- 显示简短中文错误文案
- 提供“重试”按钮

### 8.4 未接入状态

`归档` 当前只保留导航位：

- 可见
- 不承诺可进入
- 必须通过视觉弱化或短提示表达“暂未接入”

## 9. 视觉与文案约束

### 9.1 视觉

必须继续遵守现有设计系统：

- 同一套 header 节奏
- 同一套 paper / ink / moss 色系
- No-Line Rule
- 左主右辅的 desktop 布局

`回顾`页应当是首页的安静变体，而不是另一套站点。

### 9.2 文案

页面文案必须与当前能力边界一致。

允许：

- 重温旧记忆
- 再看一条
- 从已有记录里重新翻看一条

不允许：

- 每日精选
- 智能回顾
- 真随机记忆
- 算法推荐

## 10. 技术落点

### 10.1 路由层

修改位置：

- `packages/app-core/src/router/createAppRouter.tsx`

变更内容：

- 新增 `/review` 路由
- 路由切换后驱动顶部导航选中态同步

### 10.2 首页层

修改位置：

- `packages/app-core/src/screens/DesktopInbox.tsx`

变更内容：

- 顶部导航从静态文本改为可跳转结构
- `全部` 与 `回顾` 可导航
- `归档` 保持占位态

### 10.3 回顾页

新增位置：

- `packages/app-core/src/screens/DesktopReview.tsx`

职责限制：

- 发起 `review.random` 请求
- 展示加载、空、失败、成功四类状态
- 承载“换一条”按钮

不承担：

- 全局路由状态管理
- 平台桥逻辑
- 搜索逻辑
- 复杂业务缓存策略

### 10.4 API 层

直接复用现有能力：

- `apiClient.review.random()`

本次不新增：

- 新 DTO
- 新 hook 语义扩展
- 新服务端接口

## 11. 最小验收标准

1. 顶部导航可以在 `全部` 与 `回顾` 之间切换。
2. 当前所在页面具有明确选中态。
3. 进入 `/review` 后会自动展示一条旧 memo。
4. 用户点击“换一条”后会重新请求并刷新内容。
5. 页面能展示正文、标签、图片和记录时间。
6. 空状态能明确表达“还没有可回顾内容”。
7. 错误状态能允许用户重试。
8. `归档` 不会伪装成已接入能力。

## 12. 风险与约束说明

- 当前“换一条”并不保证真正随机，前端不能承诺随机体验。
- 当前接口一次只返回一条 memo，因此本页不适合设计成多卡片回顾流。
- 当前首页已经有“今天”筛选，因此不再引入“每日回顾”概念，避免能力重叠和语义误导。
